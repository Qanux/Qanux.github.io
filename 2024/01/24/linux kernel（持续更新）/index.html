可以看见ictol函数是用户与内核进行交互的函数，功能有：

  • 调用core_read函数
  • 设置off参数的值
  • 调用core_copy_func函数

可以看到core_read函数中调用了copy_to_user(a1, &v6[off], 64LL); 即将内核中的v6[off]中复制64字节到用户的a1中。由于off是我们可控的，而且程序也没有对off的合法性进行判断，所以我们可以将canary和地址等各种敏感信息泄露出来
在core_copy_func函数中有这么一段调用：qmemcpy(v3, &name, (unsigned __int16)a1); a1存在类型转换，存在整数溢出漏洞，我们因此可以将在name中写入rop链，然后copy到v3中，实现控制rip
而name的值可以在core_write中进行修改
所以我们的利用思路就很明显了，修改off->泄露地址->name中写rop链->整数溢出造成栈溢出执行rop链
这题有kernel rop和ret2usr两种打法

kernel rop

ret2usr

该打法就不做解释了，网上很多文章有讲，ret2usr就是用户态不能访问内核态的东西（对于没有SMAP/SMEP的情况下把内核指针重定向到用户空间的利用方式称为ret2usr），但是内核态可以访问用户态的东西，所以就可以利用内核的漏洞，在内核态中调用用户态的程序(函数或shellcode)。但这种方法在今天是很有局限性的，因为会有一种叫SMEP的保护机制禁止内核态执行用户态的程序，但这道题没有开SMEP

下面重点看看user_key_payload结构体

user_key_payload

该结构是用于存储密钥的payload，当我们调用key_read函数来获取payload的数据时，读取的长度由datalen这个变量来决定。所以我们可以想到利用UAF漏洞来将datalen的数值改大，导致在获取payload的数据时越界读，泄露出内核的地址
那我们要读取哪里才可以获得内核地址呢，我们可以看到上面的callback_head结构体里面存在一个函数指针，当密钥的type为user时，在payload被释放时该指针会被赋值为user_free_payload_rcu()，因此我们可以先释放掉一段payload，然后通过越界读来获取改指针的值，计算出内核的基址
需要注意的是，add_key() 会先分配一个临时的 obj1 拷贝 payload 后再分配一个 obj2 作为 user_key_payload，若我们先分配一个 obj 并释放后再调用 add_key() 则该 obj 不会直接成为 user_key_payload ，而是会在后续的数次分配中都作为拷贝 payload 的临时 obj 存在
在泄露出内核地址后，我们的要考虑的是如果劫持程序流，这里我们可以将seq_file和pt_regs结合起来利用。我们首先来看看pt_regs结构体

pt_regs

在我们执行系统调用时，内核会将所有寄存器压入内核栈上，形成该pt_regs结构体，该结构体位于内核的栈底。当我们劫持内核结构体中的某个函数指针时，在我们通过该函数指针劫持内核执行流时rsp与栈底的相对偏移通常是不变的，所以这些寄存器为我们布置ROP链条提供了可能，只要找到一条类似于add rsp, xxx; ret的gadget就能够完成ROP

方便调试模板

思路二

上面的思路一利用了pt_regs结构体，但是在内核开启了CONFIG_RANDOMIZE_KSTACK_OFFSET=y保护后该手法的利用难度会大大加大，所以这里利用了另外一种思路，即利用内核密钥管理来泄露内核地址和pipe管道来劫持控制程序流。首先给出部分pipe管道结构体源码

pipe_inode_info

pipe_buffer

当我们创建一个管道时，内核会创建一个VFS inode与一个pipe_inode_info结构体，其中pipe_inode_info->bufs指向pipe_buffer结构体数组，因此我们可以利用他来泄露出内核的堆上地址
当我们创建一个管道时，在内核中会分配一个pipe_buffer结构体数组，申请的内存总大小刚好会让内核从kmalloc-1k中取出一个object。pipe_buffer的page指针指向一个page结构体，而这个page结构体对应着一个内存页，这个内存页是pipe_buffer用来存取数据的buffer。该内存页是在第一次向pipe_buffer中写入数据的时候分配
总的来说，pipe_inode_info维护了一个pipe_buffer组成的双端队列，下面借用sky123师傅的图片

每一个pipe_buffer中的offset和len标记了pipe_buffer对应内存页中的数据，还是借用sky123师傅的图片

当我们关闭了管道的两端时，会触发pipe_buffer->pipe_buffer_operations->release这一函数指针

pipe_buf_operations

所以我们可以劫持该release指针。当程序运行到调用release指针时，rsi指针指向调用该rlease的pipe_buffer结构体，所以我们可以直接在pipe_buffer中写好rop链，利用rsi来栈迁移执行我们的rop链
现在回到题目中，由于前面我们提到过 add_key() 会先分配一个临时的 obj1 拷贝 payload 后再分配一个 obj2 作为 user_key_payload，若我们先分配一个 obj 并释放后再调用 add_key() 则该 obj 不会直接成为 user_key_payload ，而是会在后续的数次分配中都作为拷贝 payload 的临时 obj 存在。我们可以利用堆喷来实现将当前索引的obj成为user_key_payload，流程为（来自arttnba3师傅博客）：

  • 利用题目功能构建 UAF object
  • 堆喷射 user_key_payload ，UAF obj 作为拷贝 payload 的临时 obj 存在
  • kmem_cache_cpu 的 slub page 耗光，向 node 请求新的 slub page 分配 user_key_payload ，完成后 UAF obj 被释放并回到 kmem_cache_node
  • 继续堆喷 user_key_payload ，kmem_cache_cpu 的 slub page 耗光，向 node 请求新的 slub page 分配 user_key_payload
  • UAF obj 所在页面被取回，UAF obj 被分配为 user_key_payload
  • 利用题目功能再次释放 UAF obj，利用题目功能进行堆喷获取到该 obj，从而覆写 user_key_payload

我们可以另user_key_payload与pipe_inode_info占用同一个object，这样 pipe_inode_info 刚好会将 user_key_payload->datalen 改为 0xFFFF 使得我们能够继续读取数据，从而读取 pipe_inode_info 以泄露出 pipe_buffer 的地址
最后我们直接在pipe_inode_info中布置好fake pipe_buffer_operations和rop链，最后通过关闭管道的两端来劫持程序流来执行我们的rop链进而提权，这里利用了下面2个gadget相互配合来实现栈迁移：

漏洞分析

这题的漏洞是显而易见的，和CISCN-2017 babydrive一模一样，即release将buf释放后没有将buf指针置零，我们可以创建多个fd来利用该UAF漏洞。由于前面的题目都是用kernel rop来解决的，所以这一题我尝试使用modprobe_path来进行求解
首先我们要知道modprobe_path是什么东西。根据维基百科的说法，modprobe是最初由Rusty Russell编写的Linux程序，用于在Linux内核中添加可加载的内核模块。实际上，当我们在Linux内核中安装或卸载新模块时，就会执行这个程序。当我们执行具有未知文件类型的文件时，将执行存储在modprobe_path路径的程序。更准确地说，如果我们针对系统未知文件签名（魔术头）的文件调用execve()，则会产生以下调用

最终会调用call_modprobe()，该函数会有如下操作：

当我们在系统上执行文件类型未知的文件时，系统将会执行当前路径存储在modprobe_path中的任何文件。因此，我们所研究的技术就是使用任意写入原语，将modprobe_path覆盖到我们自己编写的Shell脚本的路径中，然后执行具有未知文件签名的虚拟文件。其结果将导致在系统仍处于内核模式时执行Shell脚本，从而导致root特权的任意代码执行。
所以我们的思路即为利用UAF漏洞，将被释放的buf的next指针改为指向modprobe_path，再利用UAF将modprobe_path修改为我们恶意脚本的路径，进而使我们能够以root权限来执行我们的恶意脚本，赋予flag能给普通用户读取的权限
在内核“堆基址”（page_offset_base） + 0x9d000 处存放着 secondary_startup_64 函数的地址，而我们可以从 free object 的 next 指针获得一个堆上地址，从而去猜测堆的基址，之后分配到一个 堆基址 + 0x9d000 处的 object 以泄露内核基址，这个地址前面刚好有一片为 NULL 的区域方便我们分配
